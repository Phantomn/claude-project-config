# /brainstorm - 요구사항 발견 스킬

## What

소크라테스 대화로 모호한 아이디어를 탐색하고, 구체적인 요구사항 브리프를 생성하여 `/breakdown`으로 넘긴다.

## vs /breakdown

| | `/brainstorm` | `/breakdown` |
|---|---|---|
| 입력 | 모호한 아이디어 | 알고 있는 요구사항 |
| 역할 | 발견 | 분해 |
| 출력 | 요구사항 브리프 | 실행 계획 |

## When

- "뭔가 만들고 싶은데 정확히 뭔지 모르겠어"
- 아이디어는 있지만 요구사항이 불명확할 때
- 범위/목적/사용자가 모호한 초기 기획 단계
- `/breakdown` 전에 탐색이 필요할 때

## Workflow

### Phase 1: 소크라테스 대화

아이디어를 받으면 아래 4개 영역을 **2-3라운드** 대화로 탐색한다.

한 번에 모든 질문을 쏟지 않고, **응답에 따라 파생 질문**을 선택한다.

**탐색 영역:**

```
1. 목적
   - 이 아이디어가 해결하려는 문제는?
   - 지금 이 문제를 해결하려는 이유는?

2. 사용자
   - 누가 사용하나? (본인 vs 타인)
   - 사용자의 현재 워크플로우는?

3. 범위
   - 반드시 있어야 하는 기능은?
   - 없어도 되는 기능은?

4. 제약
   - 기술 스택/환경 제약이 있나?
   - 보안/성능/규정 준수 요구사항이 있나?
```

**라운드 구조:**
- R1: 목적 + 사용자 파악
- R2: 범위 명확화 (R1 응답 기반)
- R3: 제약 및 MVP 경계 확정 (필요 시)

탐색이 충분하다고 판단되면 R3 생략 가능.

### Phase 2: 요구사항 브리프 생성

탐색 완료 후 아래 형식으로 브리프를 출력한다.

```markdown
# 요구사항 브리프: [아이디어명]

## 문제
[해결하려는 핵심 문제 1-2문장]

## 사용자
[타겟 사용자 및 주요 워크플로우]

## MVP 범위
**포함:**
- [필수 기능 1]
- [필수 기능 2]

**제외:**
- [범위 밖 기능]

## 제약
- [기술/환경/보안 제약]

## 미해결 질문
- [추가 탐색이 필요한 항목]

---
👉 다음 단계: `/breakdown`으로 구현 계획 수립
```

## Edge Cases

1. **아이디어가 너무 광범위**: 범위를 좁히는 질문 집중 ("MVP에서 꼭 필요한 단 하나는?")
2. **이미 요구사항이 명확**: "요구사항이 충분히 구체적입니다. `/breakdown`을 바로 사용하세요." 안내
3. **기술 선택 질문**: 탐색 단계에서 기술 추천 금지 — 제약 파악만
4. **라운드 3회 초과**: 추가 질문 대신 현재까지 파악된 내용으로 브리프 초안 제시 후 수정

## Integration

- `/breakdown`: 브리프 생성 후 핸드오프
- `extensions/modes/brainstorming.md`: 자연스러운 대화에서 자동 활성화되는 passive 모드 (별개)
